# Cold-Start Side Projects

Use this reference when the user gives a casual, rough, or one-line app idea.

## Fit

This workflow is best for lightweight cold-start side projects:

- simple personal tools,
- small Korean-first apps,
- toy-to-useful prototypes,
- local dashboards,
- recommendation or briefing helpers,
- single-user workflows,
- and ideas where the user does not yet know the exact product direction.

The goal is not to copy a full enterprise SDLC. The goal is to stop the most common cold-start failures:

- the agent silently chooses the product direction,
- the app is only a visual mockup,
- the first version does not respond to real input,
- the user wastes tokens rebuilding from scratch,
- or the monitor says done while the product is not usable.

## Lightweight Council

For low-risk side projects, keep Discovery Council short:

- Product Strategist: who uses it, why, and MVP scope.
- Technical Feasibility Architect: what is feasible, what needs fallback, and stack risk.
- Growth and Moat Strategist: why this is worth keeping and one small differentiation idea.
- UX Strategist: first-use flow and empty state.
- Risk Critic: privacy, false confidence, and rework risks.
- Planner: shortest safe sequence.

Each role may use 1-3 bullets. Brevity is allowed; skipping the gate is not.

## Decision Bias

Prefer directions that:

- can be built locally first,
- need few external accounts,
- have one obvious primary action,
- show useful output from current user input,
- and can be verified with 2-3 concrete scenarios.

Avoid directions that require heavy scraping, regulated advice, complex payments, multi-user auth, or hard integrations unless the user explicitly chooses that path.

## Completion Standard

For this niche, "done" means:

- the chosen Direction Lock exists,
- a small real app exists,
- at least one real input changes the visible output,
- an edge or empty state is handled,
- mock data is labeled or replaceable,
- and the Factory Harness passes.
